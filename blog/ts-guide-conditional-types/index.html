TS Guide: Conditional Types

I like to imagine TypeScript as the superhero version of JavaScript - armed with an impenetrable armor of static typing. It’s like having a trusty sidekick for your code, making sure nothing sneaky slips through the cracks. Among its nifty features, we’ve got these cool conditional types – think of them as the shape-shifters of the TypeScript world, here to make your code more flexible and fun. In my couple of years in the industry, I’ve bumped into TypeScript conditional types just once. But that one encounter was like stumbling upon a hidden treasure chest that piqued my curiosity and made me dive deeper into their mysteries. Now, I’m on a mission to unravel their secrets and share this newfound wisdom with the world. So, join me on this journey as we shed light on these rare gems and see how they can elevate your coding game.

The basic syntax for a conditional type looks like this:


  type MyConditionalType<T> = T extends SomeCondition ? TypeIfTrue : TypeIfFalse;
.

The code defines a TypeScript conditional type MyConditionalType that takes a generic type parameter T. It checks if T satisfies a condition represented by SomeCondition. If T meets this condition, it evaluates to TypeIfTrue, otherwise, it results in TypeIfFalse. In essence, this conditional type allows you to dynamically choose one of two types for T based on a Boolean condition, making it a versatile tool for creating flexible and expressive type definitions that adapt to different scenarios.

But this is all too theoretical - let’s see this bad boy in action. Ever heard of ReturnType<Type>? It’s a built-in utility type, commonly used to construct a type consisting of the return type of function Type. This is how it works, illustrated by a very simple example:


  function add(a: number, b: number): number {
    return a + b;
  }

  type AddResult = ReturnType<typeof add>;

  const result: AddResult = 5;
.

But, did you know that, under the hood, it uses conditional types?

This is how it’s implemented:


  type ReturnType<T extends (...args: any) => any> = T extends (...args: any) => infer R ? R : any;
.

Woah. That’s a lot of keywords and types. Let’s try and dissect it and see how conditional types are used here.

  1. type ReturnType<T extends (...args: any) => any>: This tells us that ReturnType is a type that takes a generic type parameter T, which must be a function type taking any number of arguments and returning any type.

  2. T extends (...args: any) => infer R: This is the first part of the conditional type. It checks whether the generic type T extends or matches the shape of a function that takes any number of arguments. If this condition is met, it proceeds to the next part.

  3. ? R : any: This is the second part of the conditional type. If the condition T extends (...args: any) => infer R is true, it infers the return type of the function (represented by infer R) and returns it as R. If the condition is false, it defaults to the type any.

Cool, right? Let’s create an example ourselves!


  type ObjectsReadonly<T> = {
    [K in keyof T]: T[K] extends object ? Readonly<T[K]> : T[K];
  };
.

So, let’s see what do we got here. ObjectReadonly is a generic mapped type. For every key of the original type T, it checks whether the value is an object, and, based on that assumption, makes all properties of these nested objects - read only. Primitive values on the other hand, stay unaltered.

Where can we use this, you may ask? The snippet below might clear that up for you!


  type Product = {
      publicId: string;
      publicSKU: string;
      isPhysical: boolean;
      privateData: {
        privateId: string;
        // ...
      }
  }

  const exampleObject: ObjectsReadonly<Product> = {
      publicId: 'public-id',
      publicSKU: 'public-sku',
      isPhysical: true,
      privateData: {
        privateId: 'private-id'
        // ...
      }
  };

  /* This is fine! Primitive values are open to changes! */
  exampleObject.publicId = 'new value';

  /* Defcon 1! All properties in nested objects are readonly! This will not work */
  exampleObject.privateData.privateId = 'new value';
.
