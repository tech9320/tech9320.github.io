TS Guide: Template Literal Types

Welcome back to our TypeScript guide! In our last episode, we explored the mysteries of conditional types, the shape-shifters of the TypeScript world. Now, get ready to embark on a journey into the realm of template literal types, where strings become dynamic templates for crafting precise and powerful types.

The Basics

Template literal types, introduced in TypeScript 4.1, extend the capabilities of string literal types. With template literal types, we can create types by concatenating and manipulating string literal types, resulting in expressive and reusable type definitions.

The basic syntax looks like this:


    type MyTemplateLiteralType<T extends string> = `Hello, ${T}!`;
.

In this example, MyTemplateLiteralType is a type that takes a string literal type T and creates a new type by concatenating it with the string literal "Hello, " and the string literal type T. This might seem trivial at first, but the real magic happens when we start using template literal types for more complex scenarios.

Dynamic Paths and Keys

One powerful application of template literal types is in creating types that represent dynamic paths or keys in objects. Consider the following example:


    type PathBuilder<T extends string> = `api/${T}/endpoint`;

    const userEndpoint: PathBuilder<"users"> = "api/users/endpoint";
    const productEndpoint: PathBuilder<"products"> = "api/products/endpoint";
.

In this case, PathBuilder is a template literal type that dynamically constructs paths for different endpoints. This not only ensures type safety but also makes it easier to refactor and maintain your code.

String Manipulation

Template literal types enable string manipulation within types, offering a level of flexibility previously unseen in TypeScript. Let’s create a type that adds an ellipsis in the middle of a string:


    type AddEllipsis<S extends string> =
    S extends `${infer Start}${infer End}`
        ? `${Start}...${End}`
        : S;

    type ShortenedGreeting = AddEllipsis<"Hello, TypeScript!">; // "Hello, Typ..."
.

In this example, AddEllipsis takes a string S and adds an ellipsis in the middle of it, creating a transformation not directly covered by existing TypeScript utility types.

Practical Applications

Let’s put template literal types to use in a real-world scenario. Imagine you have a function that fetches data from different API endpoints. With template literal types, you can create a type-safe API client:


    type ApiEndpoints = "users" | "products" | "orders";

    type ApiClient<T extends ApiEndpoints> = {
    endpoint: `api/${T}`;
    fetchData: () => Promise<any>;
    };

    const usersApiClient: ApiClient<"users"> = {
    endpoint: "api/users",
    fetchData: () => fetch("api/users").then((res) => res.json()),
    };

    const productsApiClient: ApiClient<"products"> = {
    endpoint: "api/products",
    fetchData: () => fetch("api/products").then((res) => res.json()),
    };
.

By using template literal types, we create a type-safe association between the endpoint type (T) and the corresponding API client configuration.

Conclusion

In the end, there’s a question up in the air - does this have any use in standard projects we encounter everyday?

I’ve personally tinkered around with this concept, and found one very interesting use-case that I can see being useful in a lot of places:


    type ContainsString<T extends string> = `${string}${T}${string}`;

    /* Valid */
    const a: ContainsString<"9320"> = "Tech9320 rocks!";

    /* Also valid */
    const a: ContainsString<"9320"> = "This string ends with 9320";

    /* No longer valid! */
    const a: ContainsString<"9320"> = "Nope, sorry, not gonna work";
.

It’s something I’ll definitely use in the future!
